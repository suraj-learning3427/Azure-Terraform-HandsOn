The 5 Most Important Concepts

1. Managed Identity

Never store credentials. Use Managed Identity for all Azure resource authentication. System-assigned for single resources, user-assigned for shared identity across multiple resources.

2. Remote State

Always use Azure Storage backend for Terraform in teams. State locking prevents conflicts. Separate state files per environment. Never use local state in a team.

3. Deployment Slots

Zero-downtime deployments. Slot-specific settings stay with the slot. Swap is atomic. Rollback in < 30 seconds. Canary with traffic routing before full swap.

4. DORA Metrics

Deployment Frequency, Lead Time for Changes, Change Failure Rate, Time to Restore. Elite: multiple deploys/day, <1hr lead time, <5% failure rate, <1hr recovery.

5. Shift-Left Security

Security at every stage: IDE (SonarLint) → pre-commit (gitleaks) → PR (SAST) → CI (Trivy, Checkov) → staging (DAST) → production (Defender for Cloud). Cost of fixing in dev = 10x cheaper than production.

Structure: trigger → build → test (with coverage) → security scan (Trivy, Checkov, SAST) → artifact → deploy to staging → approval → deploy to production. Key points: OIDC auth (no stored secrets), quality gates (80% coverage, 0 new vulnerabilities), rollback plan (slot swap back in <30s).

App Service deployment slots: deploy to staging → smoke test → canary 10% → approval → swap to production. Rollback: swap back in <30 seconds. Database migrations use expand-contract pattern for backward compatibility.

Deployment Frequency (how often you deploy to production), Lead Time for Changes (commit to production), Change Failure Rate (% of deployments causing incidents), Time to Restore Service (how long to recover). Elite: multiple deploys/day, <1hr lead time, <5% failure rate, <1hr recovery.

Variable groups linked to Key Vault. OIDC federated identity (no stored service principal secrets). Never store secrets in pipeline YAML. Secret scanning in CI (gitleaks, detect-secrets). Key Vault references in App Service app settings.

Blue-green: atomic switch, instant rollback, full traffic shift at once. Canary: gradual traffic shift (5%→25%→100%), validate with real traffic, automated rollback on metric breach. Blue-green for infrastructure upgrades, canary for feature releases.

State maps Terraform config to real Azure resources. Without it, Terraform can't know what already exists. Remote state (Azure Storage) enables team collaboration, state locking (prevents concurrent applies), and shared source of truth. Never use local state in a team.

for_each almost always — stable keys, removing one item only affects that resource. count has index instability — removing the middle item shifts all subsequent indexes, causing unnecessary destroy/recreate. Use count only for optional resources: count = var.enabled ? 1 : 0.

Container Apps for most microservices — removes 80% of K8s overhead, built-in scale-to-zero, Dapr, traffic splitting. AKS when you need full K8s control, GPU nodes, custom operators, or your team has deep Kubernetes expertise. For new projects, start with Container Apps.

TDE: storage-level encryption, DB engine sees plaintext — always on by default. Always Encrypted: client-side encryption, DB engine never sees plaintext, only the application with the column master key can decrypt. Use TDE always. Use Always Encrypted for most sensitive columns (SSN, credit card).

Managed Identity: Azure manages credentials automatically, you never see them, no rotation needed. Service principal: you manage credentials, must rotate, can be leaked. Always use Managed Identity when available (App Service, Functions, AKS, VMs). Use service principal only when Managed Identity isn't available.

Expand-contract pattern: Phase 1 (Expand) — add new column as nullable, both old and new code work. Phase 2 (Deploy) — deploy new code that writes to the new column. Phase 3 (Contract) — add NOT NULL constraint, remove old code paths. 3 deployments, safe rollback at each step.

Git as single source of truth for cluster state. Flux/ArgoCD continuously reconciles cluster to match Git. Every change is a git commit — full audit trail, easy rollback. CI updates image tag in Git → Flux detects change → applies to AKS automatically.

Right-size (check utilization — if CPU <30%, downsize) → Reserved Instances (35-55% savings) → scale-in during off-hours (scheduled profiles) → move batch jobs to Functions (Consumption plan) → dev/test pricing → consolidate apps onto fewer App Service Plans.

Short-lived tokens from identity provider — no stored credentials. Azure trusts GitHub/Azure DevOps OIDC issuer. No secrets to rotate or leak. Traditional service principal secrets expire, can be leaked from pipeline logs, and require manual rotation.

Separate directories per environment (environments/dev/, environments/prod/) each with own state file. Shared logic in modules/. Each environment calls the same modules with different variable values. Never use Terraform workspaces for environment isolation — they share the same backend.

Moving security checks earlier: IDE → pre-commit → PR → CI → staging → production. Cheaper to fix in development than production (10x cost difference). Tools: SonarLint (IDE), gitleaks (pre-commit), SonarCloud (SAST), Trivy (containers), Checkov (IaC), ZAP (DAST), Defender for Cloud (production).

A network interface in your VNet with a private IP that connects to an Azure service (SQL, Storage, Key Vault). The service is accessible only from your VNet — no public internet access. Required for production databases and Key Vaults. Disable public network access after creating the private endpoint.

App Service: az monitor autoscale create with CPU-based rules (scale out >70%, scale in <30%). AKS: Cluster Autoscaler for nodes + HPA for pods. VMSS: azurerm_monitor_autoscale_setting in Terraform. Always pair scale-out with scale-in rules. Use cooldown periods to prevent thrashing.

3-phase backward-compatible DB migration: Expand (add nullable column — both old and new code work) → Deploy (new code writes to new column) → Contract (add NOT NULL, remove old code paths). Ensures zero downtime and safe rollback at each phase.

Azure Front Door Premium → App Service/AKS in 2+ regions → SQL failover groups → Redis geo-replication. Front Door routes to nearest healthy region. RTO <2 minutes (Front Door detects failure in ~30s), RPO <5 seconds (SQL replication lag). Cost: running two full environments doubles compute costs.

A post-incident review that focuses on systemic causes rather than blaming individuals. Goal: identify what failed in the process/system and how to prevent recurrence. Output: timeline of events, root cause analysis, action items with owners and due dates. Psychological safety is essential.

Kubenet: pods get IPs from a separate pod CIDR, not from the VNet — limited connectivity to other VNet resources. Azure CNI: pods get IPs directly from the VNet subnet — first-class VNet citizens. Required for: private endpoints, direct pod-to-pod communication across VNets, network policies with Calico. Azure CNI requires more IP addresses — plan subnet size carefully.

1) Verify no Terraform process is actually running. 2) Get the lock ID from the error message. 3) Run terraform force-unlock <lock-id>. 4) Run terraform plan to verify state is consistent. Prevention: set pipeline timeouts so crashed pipelines release locks automatically.

Audit (log non-compliant), Deny (block creation), DeployIfNotExists (auto-remediate), Modify (auto-fix properties). Use Audit first to understand impact, then switch to Deny for hard requirements. Use DeployIfNotExists for auto-enabling diagnostic settings on new resources.

Mounts Key Vault secrets as files in pod volumes. Preferred over environment variables because: secrets are fetched at pod startup, automatically rotated, not visible in kubectl describe pod, and follow least privilege. Enable with key_vault_secrets_provider { secret_rotation_enabled = true } in AKS Terraform config.
